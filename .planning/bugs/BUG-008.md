# BUG-008: TRANSCRIPT_SESSION conversations render wrong UI at /chat/

- **Severity:** HIGH
- **Phase:** 2 (Discovery and Knowledge)
- **Status:** Open

## Description

When a TRANSCRIPT_SESSION conversation is opened at `/projects/[projectId]/chat/[conversationId]`, it is rendered using the general-purpose `ChatInterface` component with a text input box and streaming chat UI. However, transcript sessions are not interactive chat sessions -- they are background AI processing jobs where the user should see extraction results (ExtractionCards), processing status, and accept/reject controls. Instead, users see a generic chat interface with an input field, and typing a message hits the `/api/chat` route which treats it as a general chat (no TRANSCRIPT_SESSION branch in the route handler). The extraction result messages with `toolCalls.type === "extraction_results"` are also not rendered as structured cards.

## Location

- `src/app/(dashboard)/projects/[projectId]/chat/[conversationId]/page.tsx:33-38` — Maps TRANSCRIPT_SESSION to `"TASK_SESSION"`, losing the conversation type distinction
- `src/app/api/chat/route.ts:107-119` — Only handles STORY_SESSION and GENERAL_CHAT; no TRANSCRIPT_SESSION branch
- `src/components/chat/chat-interface.tsx:17` — `conversationType` union does not include TRANSCRIPT_SESSION

## Root Cause

The conversation page maps all non-GENERAL_CHAT, non-STORY_SESSION types to `"TASK_SESSION"` (line 33-38 of the page). The `ChatInterface` component only understands three types: `GENERAL_CHAT`, `TASK_SESSION`, and `STORY_SESSION`. There is no `TRANSCRIPT_SESSION` variant, so transcript processing conversations get the wrong UI -- a standard chat input instead of a read-only extraction results view with polling for status updates.

## Fix

1. Either add a `TRANSCRIPT_SESSION` variant to `ChatInterface` that renders a read-only view with `MessageList` (no input box), polls the `/api/transcripts/[transcriptId]/messages` endpoint via SWR for live updates, and renders ExtractionCards for messages with `toolCalls.type === "extraction_results"`.
2. Or redirect TRANSCRIPT_SESSION conversations to a dedicated transcript detail page (e.g., `/projects/[projectId]/transcripts/[transcriptId]`) instead of routing them through the chat page. This is the cleaner approach since transcript sessions are fundamentally different from chat sessions.
3. In either case, the conversation page at line 33-38 should explicitly handle TRANSCRIPT_SESSION rather than falling through to TASK_SESSION.
