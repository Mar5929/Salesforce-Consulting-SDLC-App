# BUG-029: Jira disconnect is a no-op placeholder

- **Severity:** LOW
- **Phase:** 5 (Documents, QA, Admin)
- **Status:** Fixed

## Description

The Jira settings section has a "Disconnect" button that opens a confirmation dialog, but the actual disconnect action is a placeholder. The `AlertDialogAction` button is visually disabled (`opacity-50 cursor-not-allowed` and `disabled` prop) and clicking it only shows a toast saying "Jira disconnect is not yet implemented." There is no `deleteJiraConfig` server action. Users who need to remove a Jira configuration cannot do so through the UI.

## Location

- `src/app/(dashboard)/projects/[projectId]/settings/jira-settings-section.tsx:57-59` -- `handleDisconnect` is a no-op that shows a toast
- `src/components/jira/jira-config-form.tsx:243-246` -- Disconnect button is explicitly disabled with `disabled` prop and dimmed styling

## Root Cause

The disconnect feature was intentionally deferred (marked with a TODO comment at line 57 of `jira-settings-section.tsx`). No `deleteJiraConfig` server action exists. The button is rendered in a disabled state, but users can still open the confirmation dialog and see the non-functional disconnect button, which is confusing.

## Fix

1. Create a `deleteJiraConfig` server action in `src/actions/jira-sync.ts` that:
   - Verifies PM role
   - Deletes the `JiraConfig` record for the project
   - Optionally retains `JiraSyncRecord` entries for audit history
   - Revalidates the settings page path

2. Wire `handleDisconnect` in `jira-settings-section.tsx` to call the new action.

3. Remove the `disabled` prop and the dimmed styling from the disconnect `AlertDialogAction` in `jira-config-form.tsx`.

4. Add `router.refresh()` in the success callback to update the UI after disconnection.

## Related Files

- `src/actions/jira-sync.ts` -- Where the new `deleteJiraConfig` action should be added
- `src/components/jira/jira-config-form.tsx` -- Contains the disabled disconnect button
