# BUG-010: Sprint board drag-and-drop has no role awareness

- **Severity:** HIGH
- **Phase:** 3 (Story Management)
- **Status:** Open

## Description

The sprint board allows any user to drag and drop story cards between columns, triggering status transitions. While the server action `updateStoryStatus` validates role permissions via `canTransition()`, the client-side board provides no feedback about which transitions are allowed for the current user's role. A BA or QA user can drag a card to any column, only to get a vague error toast after the server rejects it. Worse, the optimistic UI (router.refresh on success) gives no pre-drop indication that the operation will fail, creating a confusing experience.

## Location

- `src/components/sprints/sprint-board.tsx:247-303` -- `SprintBoard` component accepts no `userRole` prop and performs no client-side role checks
- `src/components/sprints/sprint-board.tsx:97-155` -- `BoardCard` is unconditionally `draggable` for all users
- `src/components/sprints/sprint-board.tsx:274-288` -- `handleDrop` sends the transition to the server with no pre-validation

## Root Cause

The `SprintBoardProps` interface (line 37) only accepts `stories` and `projectId` -- it has no `userRole` prop. The `BoardCard` component always sets `draggable` to true. The `handleDrop` function does not call `canTransition()` client-side before invoking the server action. The status machine module (`src/lib/story-status-machine.ts`) exports `canTransition` and `getAvailableTransitions` which could be used client-side, but neither is imported.

## Fix

1. Add `userRole` to the `SprintBoardProps` interface in `src/components/sprints/sprint-board.tsx`.

2. Import `canTransition` from `@/lib/story-status-machine` (it's a pure function, works client-side).

3. In `BoardCard`, conditionally set `draggable` based on whether the user's role has any valid transitions from the story's current status (use `getAvailableTransitions`). For roles with no transitions (BA, QA), set `draggable={false}` and change cursor to `cursor-default`.

4. In `handleDrop`, before calling `executeUpdateStatus`, check `canTransition(currentStatus, targetStatus, userRole)`. If invalid, show a descriptive toast like "Your role cannot move stories to this status" and return early.

5. Optionally, during `dragOver`, visually indicate valid vs. invalid drop targets by checking the dragged story's status against each column's status using `canTransition`.

6. Update the parent component that renders `SprintBoard` to pass the `userRole` prop.

## Related Files

- `src/lib/story-status-machine.ts` -- Exports `canTransition` and `getAvailableTransitions`, the functions to use for client-side gating
- `src/actions/stories.ts:267-334` -- Server-side `updateStoryStatus` action that already validates roles (the server backstop)
- `src/components/sprints/sprint-dashboard.tsx` -- Likely parent component that should pass `userRole` to `SprintBoard`
