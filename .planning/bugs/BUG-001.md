# BUG-001: proxy.ts never runs as Next.js middleware — all auth silently disabled

- **Severity:** CRITICAL
- **Phase:** 1 (Foundation)
- **Status:** Fixed

## Description

Next.js only auto-discovers middleware from a file named `middleware.ts` (or `.js`) at the project root or `src/` directory. The file is currently named `src/proxy.ts`, which Next.js ignores entirely. This means:

- Clerk auth enforcement never runs — unauthenticated users can access protected routes
- The `x-pathname` header is never injected into requests
- `projectId` extraction in the dashboard layout always returns `null`
- `currentMemberRole` is always `undefined` — role-based sidebar gating is broken
- `questionReviewCount` and `openDefectCount` are always `undefined` — sidebar badges never show

The commit `16255e6` ("fix(build): remove middleware.ts conflicting with proxy.ts") caused this by removing the correctly-named file.

## Location

- `src/proxy.ts` — the middleware file with the wrong name
- `src/app/(dashboard)/layout.tsx:18` — reads `x-pathname` header that is never set

## Root Cause

Next.js middleware filename convention requires `middleware.ts`. The file was renamed to `proxy.ts` to resolve a build conflict, but `proxy.ts` is not a recognized middleware filename in any Next.js version.

## Fix

Rename `src/proxy.ts` to `src/middleware.ts`. If there was a build conflict with the previous `middleware.ts`, the conflict must be resolved in the file contents, not by renaming.

## Related Files

- `src/app/(dashboard)/layout.tsx` — depends on `x-pathname` header
- `src/components/layout/app-shell.tsx` — receives role/badge data from layout
- `src/components/layout/sidebar.tsx` — renders badges and role-gated nav items
