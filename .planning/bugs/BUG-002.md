# BUG-002: Embedding pipeline silently skips all work — payload shape mismatch

- **Severity:** CRITICAL
- **Phase:** 2 (Knowledge Brain)
- **Status:** Fixed

## Description

Every `EMBEDDING_BATCH_REQUESTED` event is fired with `{projectId, sourceEntityType, sourceEntityId}` but the consumer (`embeddingBatchFunction`) expects `{projectId, entities: EmbeddingEntity[]}`. Since `entities` is always `undefined`, the function hits the guard `if (!entities || entities.length === 0)` and returns `{status: "skipped"}`. No embeddings are ever generated, which means semantic search returns zero results for all content.

This affects every caller of `EMBEDDING_BATCH_REQUESTED`:
- `transcript-processing.ts` (after transcript extraction)
- `staleness-detection.ts` (after article bootstrap)
- `article-refresh.ts` (after article content update)

## Location

- `src/lib/inngest/functions/transcript-processing.ts:293-299` — fires event with wrong shape
- `src/lib/inngest/functions/embedding-batch.ts:83-89` — expects `entities` array
- `src/lib/inngest/functions/staleness-detection.ts` — also fires with wrong shape
- `src/lib/inngest/functions/article-refresh.ts` — also fires with wrong shape

## Root Cause

The event producer and consumer were built in different plans without a shared type contract for the event payload.

## Fix

**Option A (recommended):** Change `embedding-batch.ts` to accept `{sourceEntityType, sourceEntityId}` and resolve the entities to embed internally (look up the knowledge articles, org components, etc. based on the source).

**Option B:** Change all callers to resolve the `entities` array before firing the event — requires each caller to know the embedding entity format.
