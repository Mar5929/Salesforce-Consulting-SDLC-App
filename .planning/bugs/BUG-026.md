# BUG-026: OAuth CSRF error redirects to root with unhandled param

- **Severity:** LOW
- **Phase:** 4 (Salesforce Org Connectivity)
- **Status:** Open

## Description

When the Salesforce OAuth callback encounters a CSRF validation failure (invalid or expired state token), the user is redirected to `/?error=oauth_failed` (the application root) instead of back to the project's org settings page. This happens because the `projectId` is only available after successful state validation, so the redirect on line 36 cannot include it. The root page does not handle the `error=oauth_failed` query parameter, so the user sees the dashboard with no error message or context about what went wrong.

## Location

- `src/app/api/auth/salesforce/callback/route.ts:36` â€” `return NextResponse.redirect(\`${APP_URL}?error=oauth_failed\`)`
- `src/app/api/auth/salesforce/callback/route.ts:29-31` â€” Error/missing code/state also redirects to root

## Root Cause

There are two redirect-to-root paths in the callback handler:

1. **Line 29-31**: If Salesforce returns an error, or `code`/`state` params are missing, the redirect goes to `/?error=oauth_failed`. The `projectId` is unavailable because `state` hasn't been parsed yet.

2. **Line 35-36**: If `validateOAuthState` returns null (token not found, expired, or already consumed), the redirect goes to `/?error=oauth_failed`. The `projectId` was stored in the now-deleted/expired state record and cannot be recovered.

In both cases, the root page (`/` or the dashboard) does not check for the `error=oauth_failed` query parameter, so no error UI is shown.

## Fix

**Short-term (handle the param at root):**
1. In the main dashboard page or layout, check for `searchParams.error === "oauth_failed"` and display an alert/toast explaining that the Salesforce authorization failed and the user should try again from their project's org settings.

**Better approach (preserve projectId where possible):**
1. For the state validation failure on line 36: Before deleting the state token in `validateOAuthState`, first attempt to read the `projectId` from it. Modify `validateOAuthState` to return `{ projectId }` even when expired, so the callback can redirect to the correct project page with the error.
2. For the missing params path on lines 29-31: The `projectId` is genuinely unavailable. Consider encoding the `projectId` in the OAuth state token in a way that can be extracted even before full validation (e.g., prefix the state with `projectId:token`), or accept the root redirect and ensure the root page handles the error param.
