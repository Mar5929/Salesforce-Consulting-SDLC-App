# BUG-015: Role-based nav has no restrictions on Phase 2+ features

- **Severity:** MEDIUM
- **Phase:** 1 (Foundation)
- **Status:** Open

## Description

The sidebar navigation displays all Phase 2-5 feature links (Chat, Knowledge, Dashboard, Work, Backlog, Sprints, Org, Analysis, Documents, Defects, PM Dashboard) to every user once a project is selected, regardless of their role. The `roles` array that controls visibility is only set on 3 items (Documents, PM Dashboard, Settings/Team -- restricted to PM and SOLUTION_ARCHITECT). All other nav items have `roles: undefined`, which means they are visible to every role. Additionally, there is no phase-based gating -- features from phases not yet built appear as navigable links that would lead to 404 pages or empty shells.

## Location

- `src/components/layout/sidebar.tsx:27-162` -- `buildNavItems()` function: only Documents (line 123), PM Dashboard (line 141), Settings (line 150), and Team (line 159) define a `roles` array. All other items (Questions, Transcripts, Chat, Knowledge, Dashboard, Work, Backlog, Sprints, Org, Analysis, Defects) have no role restrictions.
- `src/components/layout/sidebar.tsx:179-181` -- `visibleItems` filter: only checks `item.roles` for visibility, has no phase-based gating logic.

## Root Cause

The sidebar was built with the full nav structure for all 5 phases, but role-based access was only added to a few PM/SA-specific items. There is no concept of "phase gating" or "feature flags" in the nav -- every item that lacks a `roles` array is shown to all authenticated project members. The `NavItem` interface supports `roles` and `disabled` but has no `phase` or `featureFlag` property.

## Fix

Two separate concerns need addressing:

**1. Add role restrictions to role-sensitive nav items:**

In `buildNavItems()` in `src/components/layout/sidebar.tsx`, add appropriate `roles` arrays to items that should be restricted. For example:
- "Sprints" -- PM, SOLUTION_ARCHITECT (sprint management is a PM function)
- "Org" / "Analysis" -- SOLUTION_ARCHITECT, DEVELOPER (Salesforce org access)

The exact role mappings should be derived from the PRD (Section on role permissions).

**2. Add phase-based gating to prevent showing unbuilt features:**

Option A (simpler): Add a `phase` property to `NavItem` and a `currentPhase` constant (or environment variable) that controls which phases are enabled. Filter out items whose phase exceeds the current build phase.

Option B (feature flags): Add a `featureFlag` property to `NavItem` and gate visibility behind a config object or environment variable like `NEXT_PUBLIC_ENABLED_FEATURES`.

Recommended approach for V1: Add a `phase` number to each `NavItem` and filter in `visibleItems`:
```ts
interface NavItem {
  label: string
  icon: LucideIcon
  href: string
  roles?: string[]
  badge?: number
  disabled?: boolean
  phase?: number  // Build phase when this feature ships
}

const CURRENT_PHASE = 1  // Update as phases are completed

const visibleItems = NAV_ITEMS.filter(
  (item) =>
    (!item.phase || item.phase <= CURRENT_PHASE) &&
    (!item.roles || (currentMemberRole && item.roles.includes(currentMemberRole)))
)
```
