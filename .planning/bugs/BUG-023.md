# BUG-023: Document generation progress has polling race condition

- **Severity:** MEDIUM
- **Phase:** 5 (Documents, QA, Admin)
- **Status:** Fixed

## Description

The `GenerationProgress` component polls for document completion by comparing document counts and checking for a recent document matching the template ID. There is a race condition: the first poll sets `initialDocCountRef` to the current count and returns early (skipping completion check). If the Inngest document generation function completes fast enough that the new document already exists on the first poll, the component will record the "with new document" count as the baseline and never detect completion, leaving the user stuck at a fake progress bar until the 2-minute timeout.

## Location

- `src/components/documents/generation-progress.tsx:48-50` -- First poll records initial count and returns early without checking for the new document
- `src/components/documents/generation-progress.tsx:119` -- `fetchDocuments` in the useEffect dependency array is unstable (new ref each render from `useAction`), risking interval restart on every render

## Root Cause

Two issues combine:

1. **First-poll blind spot:** The logic assumes the document does not exist yet on the first poll. If generation is fast (or the component mounts late), the new document is already present when `initialDocCountRef` is first set, so `data.length > initialDocCountRef.current` is never true on subsequent polls.

2. **Unstable callback reference:** `fetchDocuments` from `useAction` creates a new function reference on each render. Including it in the `useEffect` dependency array causes the interval to be cleared and recreated on every render cycle, resetting `pollCountRef` and potentially missing completion events.

## Fix

1. **Fix the first-poll blind spot:** Instead of recording initial count and returning early, record a timestamp when generation was requested (passed as a prop or captured on mount). On every poll including the first, check for a document with `templateUsed === templateId` and `createdAt > generationStartTime`. This eliminates the count-comparison approach entirely.

2. **Stabilize the callback:** Wrap `fetchDocuments` usage in a `useRef` or use `useCallback` to maintain a stable reference. Remove `fetchDocuments` from the `useEffect` dependency array (the eslint-disable is already partially in place but not correctly applied).

3. **Alternative:** Have the generation action return a `documentId` to poll for directly, which eliminates the need for heuristic matching entirely.
