# BUG-045: Analysis tab Prisma error — isArchived unknown on conversation.findMany()

- **Severity:** HIGH
- **Phase:** 2 (Discovery and Knowledge Brain)
- **Status:** Fixed

## Description

Clicking the Analysis tab produces a Prisma error:

```
Invalid `prisma.conversation.findMany()` invocation:
{
  where: {
    isArchived: false,
    projectId: "l15ie8bsdib4nhrthwa2w5rl",
    ...
  }
}
Unknown argument `isArchived`. Available options are marked with ?.
```

The error indicates `isArchived` is not a recognized field on the Conversation model's generated Prisma client, even though the field exists in the schema.

## Location

- `src/app/(dashboard)/projects/[projectId]/org/analysis/page.tsx` — Analysis tab page
- `src/actions/conversations.ts:161` — `getConversations` query that uses `isArchived` filter
- `prisma/schema.prisma:1066` — `isArchived Boolean @default(false)` field definition

## Root Cause

The `isArchived` field IS defined in the Prisma schema (line 1066 of `schema.prisma`), but the generated Prisma client doesn't recognize it. This typically means:
1. The Prisma client hasn't been regenerated after the `isArchived` field was added (`npx prisma generate` was not run)
2. Or a database migration hasn't been applied (`npx prisma migrate dev`)
3. Or the deployed version's Prisma client is stale relative to the schema

The Analysis tab page itself may not query conversations directly — it likely calls a shared action like `getConversations` which filters on `isArchived`, and this query fails because the generated client is out of date.

## Fix

1. Run `npx prisma generate` to regenerate the Prisma client from the current schema
2. Run `npx prisma migrate dev` if the migration for `isArchived` hasn't been applied
3. Verify the analysis page's data fetching path to confirm which action triggers the conversation query
4. Redeploy after regenerating the client
