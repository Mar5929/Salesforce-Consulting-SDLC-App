# BUG-016: Transcript messages API doesn't validate ownership

- **Severity:** MEDIUM
- **Phase:** 2 (Discovery and Knowledge)
- **Status:** Open

## Description

The `GET /api/transcripts/[transcriptId]/messages` endpoint verifies project membership and that the conversation belongs to the project, but it never verifies that the `transcriptId` in the URL path actually belongs to the specified `projectId` or that the conversation is linked to that transcript. An authenticated project member could poll messages for any conversation in the project by passing an arbitrary `conversationId` query param, regardless of which transcript the URL references. The `transcriptId` path parameter is completely ignored after extraction.

## Location

- `src/app/api/transcripts/[transcriptId]/messages/route.ts:22-54` — The `transcriptId` is destructured from params (line 22) but never used in any query or validation

## Root Cause

The route extracts `transcriptId` from the URL path but only uses the `projectId` and `conversationId` query parameters for its database lookup. There is no check that the transcript exists, belongs to the project, or is associated with the given conversation. The transcript ID serves no functional purpose in the current implementation -- the endpoint is effectively a general conversation messages endpoint disguised as a transcript-scoped route.

## Fix

1. Add a query to verify the transcript exists and belongs to the project:
```typescript
const transcript = await prisma.transcript.findFirst({
  where: { id: transcriptId, projectId },
})
if (!transcript) {
  return NextResponse.json([], { status: 404 })
}
```

2. Verify the conversation is linked to this transcript. Since there is no FK (see BUG-006), this requires either:
   - Fixing BUG-006 first to add a proper transcript-conversation FK, then checking that the conversation's `transcriptId` matches
   - Or checking the conversation title matches the `"Processing: ${transcript.title}"` pattern (fragile but matches current behavior)

3. Consider removing `transcriptId` from the URL if the endpoint is really just a conversation messages endpoint, and rename the route to `/api/conversations/[conversationId]/messages`.
