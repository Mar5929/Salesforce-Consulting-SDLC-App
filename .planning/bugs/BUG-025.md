# BUG-025: Revisited STORY_SESSION conversations broken — epicId URL-only

- **Severity:** MEDIUM
- **Phase:** 7 (Story Generation E2E Wiring)
- **Status:** Fixed

## Description

When a user revisits a previously created STORY_SESSION conversation (e.g., from the conversations list), the `epicId` and `featureId` are not available because they were only passed as URL search params during the initial redirect. The Conversation model has no metadata field to persist these values. Without `epicId`, the chat API route at line 109 falls through to the general chat prompt instead of the story generation prompt, and the `create_story_draft` tool still appears but has no epic/story context to work with. This means revisited story sessions lose all story-generation intelligence and behave like a broken general chat.

## Location

- `src/actions/conversations.ts:217-239` — `initiateStorySession` returns `epicId`/`featureId` for URL params but does not persist them on the Conversation record
- `src/app/(dashboard)/projects/[projectId]/chat/[conversationId]/page.tsx:9,22` — Reads `epicId`/`featureId` from `searchParams`, which are empty on revisit
- `src/app/api/chat/route.ts:109` — `if (conversation.conversationType === "STORY_SESSION" && epicId)` gate fails when `epicId` is missing, falling back to general chat prompt
- `prisma/schema.prisma:1038-1055` — Conversation model has no `metadata` or `epicId` field

## Root Cause

The `initiateStorySession` action was designed to pass `epicId` and `featureId` as URL search params on the initial redirect to the chat page. The Conversation model has no metadata/JSON field to store session-scoped parameters. When a user navigates back to the conversation later (e.g., from the conversations list), the URL lacks these search params, so `epicId` is `undefined`, the story session prompt is skipped, and the AI loses its story-generation context.

## Fix

1. **Add a `metadata` JSON field to the Conversation model** in `prisma/schema.prisma`:
   ```
   metadata Json?
   ```
   Run a migration after adding the field.

2. **Persist epicId/featureId in `initiateStorySession`** (`src/actions/conversations.ts`): Store `{ epicId, featureId }` in the new `metadata` field when creating the STORY_SESSION conversation.

3. **Read metadata in the chat page** (`src/app/(dashboard)/projects/[projectId]/chat/[conversationId]/page.tsx`): After loading the conversation, extract `epicId`/`featureId` from `conversation.metadata` as a fallback when search params are missing:
   ```ts
   const resolvedEpicId = epicId ?? (conversation.metadata as any)?.epicId
   const resolvedFeatureId = featureId ?? (conversation.metadata as any)?.featureId
   ```

4. **Read metadata in the chat API route** (`src/app/api/chat/route.ts`): Similarly fall back to `conversation.metadata?.epicId` when the request body lacks `epicId`.

5. **Update the conversations list** to link to STORY_SESSION conversations with the correct search params, or rely entirely on metadata so no search params are needed.

## Related Files

- `src/components/chat/chat-interface.tsx` — Passes `epicId`/`featureId` to the transport body; would benefit from metadata fallback
- `src/lib/agent-harness/context/stories-context.ts` — Consumes `epicId` to build story context; returns empty context without it
- `src/app/api/chat/route.ts:21-59` — `buildStorySessionPrompt` requires `epicId` to function
