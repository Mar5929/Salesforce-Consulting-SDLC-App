# BUG-006: Transcript-conversation linked by fragile title match

- **Severity:** HIGH
- **Phase:** 2 (Discovery and Knowledge)
- **Status:** Fixed

## Description

When fetching a transcript's linked conversation, the `getTranscript` action finds the conversation by matching the conversation title with `startsWith: "Processing: ${transcript.title}"`. This is a fragile string-based lookup that will break if the transcript title is edited after upload, if two transcripts share the same title prefix, or if the title format changes. The result is that transcript detail pages may show the wrong conversation or no conversation at all.

## Location

- `src/actions/transcripts.ts:177-184` — `getTranscript` uses `title: { startsWith: ... }` to find the linked conversation instead of a stable foreign key

## Root Cause

The `Conversation` model has no `transcriptId` foreign key. When `uploadTranscript` creates the conversation (line 92-104), it sets the title to `"Processing: ${transcript.title}"` but does not store the transcript ID on the conversation record. Later, `getTranscript` must reverse-engineer this link by string-matching the title. There is no stable relational link between the two records.

## Fix

1. Add a nullable `transcriptId` field to the `Conversation` model in `prisma/schema.prisma` (or use the existing `sessionLogId` pattern to store the transcript reference in a metadata-like field).
2. In `uploadTranscript` (line 92-104 of `src/actions/transcripts.ts`), store `transcriptId` on the created conversation.
3. In `getTranscript` (line 177-184), replace the `title: { startsWith }` query with `where: { projectId, transcriptId }`.
4. Run `prisma migrate dev` to apply the schema change.

Alternatively, if adding a schema field is too heavy for now, the conversationId is already passed to the Inngest event and could be stored on the Transcript record itself (a `conversationId` field on Transcript). This avoids the reverse lookup entirely -- `getTranscript` would just follow the FK.

## Related Files

- `src/actions/transcripts.ts` — Contains both `uploadTranscript` (creates the link) and `getTranscript` (reads the link)
- `src/lib/inngest/functions/transcript-processing.ts` — Receives `conversationId` in event data, confirming the ID is available at creation time
- `prisma/schema.prisma` — `Conversation` model (line 1038) and `Transcript` model lack a direct FK relationship
