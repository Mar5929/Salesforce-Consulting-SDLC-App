# BUG-046: Picklist dropdowns show raw API values instead of human-readable labels

- **Severity:** MEDIUM
- **Phase:** 1 (Foundation and Data Layer)
- **Status:** Open

## Description

Across the app, several dropdown/select fields display raw API enum values (e.g., `build_phase`, `MANAGED_SERVICES`) instead of human-readable labels (e.g., "Build Phase", "Managed Services"). Specifically, the engagement type dropdown in project settings shows a nice label when the dropdown is open (e.g., "Build Phase"), but after selection, the displayed value reverts to the raw API value (e.g., `build_phase`).

## Location

- `src/components/projects/project-settings-form.tsx:23-28` — Engagement type options with label mapping
- Various components across the app with Select/dropdown components

## Root Cause

The Select component's `SelectValue` displays the `value` prop of the selected `SelectItem`, not its display label. When a `SelectItem` has `value="BUILD_PHASE"` and label text "Build Phase", the dropdown menu shows "Build Phase" (the label), but the `SelectValue` placeholder shows the raw `value` after selection.

This is a common issue with Radix UI / shadcn Select component — the `SelectValue` component needs to render the display text, not the form value. Some implementations correctly map values to labels for display, while others let the raw value leak through.

## Fix

1. Audit all Select/dropdown components across the app for raw value display
2. For each Select component, ensure the `SelectValue` displays the human-readable label, not the raw enum value
3. Create a shared utility function (e.g., `formatEnumLabel(value)`) that converts `SNAKE_CASE` enum values to human-readable format ("Snake Case")
4. Apply consistently to: engagement type, project phase, status fields, priority fields, and any other enum-backed dropdowns
5. Also check Badge components and read-only displays that render enum values
