# BUG-019: Sprint board SPRINT_PLANNED creates impossible transitions

- **Severity:** MEDIUM
- **Phase:** 3 (Story Management)
- **Status:** Fixed

## Description

When a story has status `SPRINT_PLANNED`, the sprint board maps it to the "Draft" column via `statusToColumn()`. When a user drags that card to another column (e.g., "Ready"), `columnToStatus()` returns `"READY"` as the target status. The server then evaluates `canTransition("SPRINT_PLANNED", "READY")` which is valid. However, if the user drags the card to "In Progress", the transition `SPRINT_PLANNED -> IN_PROGRESS` is attempted, which is also valid per the status machine. The real problem is the reverse: if a user drags a story from the "Draft" column (which is actually `SPRINT_PLANNED`) back to... "Draft", `columnToStatus("DRAFT")` returns `"DRAFT"`, causing the server to evaluate `canTransition("SPRINT_PLANNED", "DRAFT")` -- a valid transition that demotes the story back to actual `DRAFT` status, losing the sprint assignment context. The user intended no change (same column), but because `statusToColumn` collapsed two statuses into one column and `columnToStatus` always maps back to `DRAFT`, the drop-on-same-column check on line 281 prevents this specific case. The deeper issue is that any `SPRINT_PLANNED` story dragged from "Draft" to "Ready" will transition to `READY` (losing planned status) when the valid forward path should be `SPRINT_PLANNED -> IN_PROGRESS`.

## Location

- `src/components/sprints/sprint-board.tsx:74-76` -- `statusToColumn()` maps `SPRINT_PLANNED` to `"DRAFT"` column
- `src/components/sprints/sprint-board.tsx:79-82` -- `columnToStatus()` always returns the column key directly, so `"DRAFT"` column maps to `"DRAFT"` status, not `"SPRINT_PLANNED"`
- `src/lib/story-status-machine.ts:17` -- `SPRINT_PLANNED` allows transitions to `["IN_PROGRESS", "READY"]`, but the board's column mapping makes "Ready" too easy to reach and hides the correct forward path

## Root Cause

The sprint board maps `SPRINT_PLANNED` stories into the "Draft" column but has no mechanism to map them back correctly. `columnToStatus("DRAFT")` returns `"DRAFT"`, not `"SPRINT_PLANNED"`. This means:

1. A `SPRINT_PLANNED` story appears in "Draft" but any drag-and-drop treats it as if it were `DRAFT`, ignoring its real status.
2. The `handleDrop` function uses `currentStatus` from the drag data (which is the real `SPRINT_PLANNED`), but the target status comes from `columnToStatus(targetColumn)` which returns `"DRAFT"` for the Draft column -- creating a `SPRINT_PLANNED -> DRAFT` demotion the user did not intend.
3. There is no `SPRINT_PLANNED` column on the board, so users cannot see or interact with this status correctly.

## Fix

Two options, pick one:

**Option A (Recommended): Add a dedicated "Sprint Planned" column.**
1. Add `{ key: "SPRINT_PLANNED", label: "Sprint Planned" }` to the `COLUMNS` array between "Ready" and "In Progress".
2. Remove the `SPRINT_PLANNED -> DRAFT` mapping from `statusToColumn()` -- let each status map to its own column.
3. `columnToStatus()` already returns the column key directly, so `"SPRINT_PLANNED"` column maps to `"SPRINT_PLANNED"` status correctly.

**Option B: Fix `columnToStatus` to be context-aware.**
1. Change `handleDrop` to use the story's real `currentStatus` when the target column is `"DRAFT"`. If `currentStatus === "SPRINT_PLANNED"` and `targetColumn === "DRAFT"`, treat it as a no-op (story is already in that visual column).
2. This is fragile and confusing -- Option A is cleaner.
