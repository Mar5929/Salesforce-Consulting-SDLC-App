    1. Create `src/actions/transcripts.ts` using next-safe-action:
       - `uploadTranscript(projectId, content, title?, sourceType: 'UPLOAD' | 'PASTE')`:
         a. Validate content length (max ~10K words, roughly 50K chars per UI-SPEC error state)
         b. Create Transcript record in DB with status PENDING
         c. Create a TASK_SESSION conversation for this transcript processing
         d. Send Inngest event TRANSCRIPT_UPLOADED with { projectId, transcriptId, conversationId }
         e. Return { transcriptId, conversationId } so UI can navigate to the chat session
       - `getTranscripts(projectId)`: List transcripts with status, creation date, extracted item counts
       - `getTranscript(transcriptId, projectId)`: Fetch transcript with linked conversation and extracted items

    2. Create `src/lib/inngest/functions/transcript-processing.ts` per D-11, TRNS-05:
       - Inngest step function following existing audit-log.ts pattern
       - id: "transcript-processing"
       - concurrency: { limit: 1, scope: "env", key: "event.data.projectId" } (one transcript at a time per project)
       - retries: 3
       - event: EVENTS.TRANSCRIPT_UPLOADED
       - Steps:
         a. `step.run("load-transcript")`: Load transcript content + project context
         b. `step.run("extract-items")`: Call `executeTask(transcriptProcessingTask, { userMessage: transcriptContent, entityId: transcriptId }, projectId, userId)`. This runs the agent loop that calls create_question, create_decision, create_requirement, create_risk tools.
         c. `step.run("update-status")`: Update transcript status to COMPLETED. Save extracted item counts.
         d. `step.run("save-to-conversation")`: Write extraction results as ChatMessage records in the linked conversation so the user sees them in the chat session UI. Each extraction result is saved as a ChatMessage with role="assistant" and a structured `metadata` JSON field: `{ type: "extraction_results", groups: [{ type: "question", items: [...] }, { type: "decision", items: [...] }, ...] }`. This structured metadata is what the ExtractionCards component reads to render grouped cards (see Task 2).
         e. `step.sendEvent("trigger-downstream")`: Emit ENTITY_CONTENT_CHANGED and EMBEDDING_BATCH_REQUESTED events for newly created entities.
       - Error handling: On failure, update transcript status to FAILED. Save error to conversation as ChatMessage with `metadata: { type: "error", message: "..." }`.
       - Each step.run() is a checkpoint — Vercel function timeout safe (each step completes within 10s).

    3. Register the new function in `src/app/api/inngest/route.ts`
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User upload -> DB | Raw transcript text stored in database |
| Transcript -> AI prompt | Full transcript content sent to Claude for extraction |
| AI extraction -> DB | AI-created entities (questions, decisions, risks) written to DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-17 | Denial of Service | uploadTranscript | mitigate | Content length validation (10K word limit). Inngest concurrency limit (1 per project). |
| T-02-18 | Tampering | transcript content | mitigate | Transcript stored as-is. AI extraction goes through sanitizeToolInput before DB writes. |
| T-02-19 | Information Disclosure | Inngest function | mitigate | All DB queries use scopedPrisma. Transcript processing scoped to single project. |
| T-02-20 | Denial of Service | Inngest step function | mitigate | Each step.run() completes within Vercel 10s timeout. 3 retry limit. Concurrency limit per project. |
</threat_model>
