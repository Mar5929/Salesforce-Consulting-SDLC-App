## Tool Result Card Design Contract

### Layout Pattern (all card variants)

```
+-------------------------------------------------------+
| [Icon] Entity Type Label            [Status Badge]    |
+-------------------------------------------------------+
| Field Label: Value                                     |
| Field Label: Value                                     |
| Field Label: Value (up to 4 fields shown)              |
+-------------------------------------------------------+
| [Link to entity page]    [Timestamp if applicable]     |
+-------------------------------------------------------+
```

- Card uses `bg-muted/50` background with `border border-border` and `rounded-lg`
- Internal padding: 16px (`md`)
- Icon size: 16px (w-4 h-4), color `text-muted-foreground`
- Entity type label: 13px semibold
- Field labels: 13px regular `text-muted-foreground`
- Field values: 14px regular `text-foreground`
- Entity name/title: 14px semibold, clickable as `text-primary hover:underline` link to entity page
- Status badge uses existing `Badge` component variant matching entity status

### Query Result Card (EntityQueryCard)

- Header: "[N] [entity type plural]" (e.g., "5 Stories") at 13px semibold
- Each row: entity display ID + title + status badge, separated by `Separator`
- Maximum 10 rows displayed. If more, show "and N more..." footer text at 13px `text-muted-foreground`
- Rows are not interactive beyond the entity name link

### Mutation Confirm Card (MutationConfirmCard)

- Header: "Created [Entity Type]" or "Updated [Entity Type]" at 13px semibold
- Icon: `Check` (lucide) in `text-chart-2` (green, hsl 160 60% 45%) for creates, `Pencil` for updates
- Shows entity title + 2-3 key changed fields
- Single row, no separator needed

### Delete Confirmation Card (DeleteConfirmationCard) â€” D-05, D-06

- Background: `bg-destructive/5` (very faint red tint)
- Border: `border-destructive/20`
- Header: "Delete [Entity Type]?" at 14px semibold
- Body: "This will permanently delete [entity display name]. This action cannot be undone." at 14px regular
- Two buttons right-aligned:
  - "Keep It" â€” `variant="outline"` Button, 13px
  - "Confirm Delete" â€” `variant="destructive"` Button, 13px
- On "Confirm Delete" click: execute the actual deletion server action, then replace card with "Deleted [entity name]" confirmation text
- On "Keep It" click: replace card with "Deletion cancelled." text at 13px `text-muted-foreground`
- This card is the hard-coded structural gate. The tool returns a `pending_confirmation` state; the actual delete only fires on user button click.

### Tool Loading Indicator

- Inline in message flow, not a modal
- Shows: `Spinner` (animated) + "Running [tool_display_name]..." at 13px `text-muted-foreground`
- Spinner: 16px, `animate-spin`, `text-muted-foreground`
- Appears when tool invocation `state === "partial-call"` or `state === "call"` and no result yet

### Tool Error Card

- Background: `bg-destructive/5`
- Border: `border-destructive/20`
- Icon: `AlertCircle` (lucide) 16px `text-destructive`
- Error message: 14px regular
- No retry button on the card itself â€” the AI will explain the error and suggest alternatives in its next text response (D-14)

## Interaction Contracts

### Tool Execution Flow (user perspective)

1. User sends message requesting an action (e.g., "Show me all open questions")
2. AI response streams text. When tool call begins, `ToolLoadingIndicator` appears inline
3. Tool executes server-side, result streams back
4. `ToolLoadingIndicator` replaced by appropriate result card (`EntityQueryCard`, `MutationConfirmCard`, etc.)
5. AI continues with text explanation after the card

### Deletion Flow (D-05 structural gate)

1. User requests deletion (e.g., "Delete story STR-042")
2. AI calls delete tool, which returns `pending_confirmation` status (NOT the actual deletion)
3. `DeleteConfirmationCard` renders with entity details and Confirm/Keep It buttons
4. User clicks "Confirm Delete" -> client calls server action -> card updates to "Deleted" state
5. User clicks "Keep It" -> card updates to "Cancelled" state, AI acknowledges
6. If user ignores the card and sends a new message, the pending deletion expires (no action taken)

### Context Panel Live Updates (D-17)

- When a mutation tool completes (create, update, delete), call `mutate()` on relevant SWR keys
- Context panel data refreshes without page reload
- No visual indicator needed for the revalidation itself â€” the panel content simply updates

### Multi-step Workflow Display (D-16)

- When AI executes multiple tool calls in sequence, each gets its own loading indicator and result card
- Cards stack vertically in message flow with 8px gap between them
- If AI approaches the 15-call limit, it includes a text warning. No UI-level enforcement needed â€” the limit is in the API route.

## State Catalog

### Tool Invocation States

| State | Visual Treatment |
|-------|-----------------|
| `partial-call` | ToolLoadingIndicator with spinner |
| `call` (no result) | ToolLoadingIndicator with spinner |
| `result` (success) | Appropriate result card (EntityQueryCard, MutationConfirmCard, etc.) |
| `result` (error) | ToolErrorCard with error details |
| `result` (pending_confirmation) | DeleteConfirmationCard with Confirm/Keep It buttons |

### Delete Confirmation States

| State | Visual Treatment |
|-------|-----------------|
| Pending | DeleteConfirmationCard with active buttons |
| Confirmed | Card replaced with "Deleted [name]." success text |
| Cancelled | Card replaced with "Deletion cancelled." muted text |
| Expired (user navigated away) | No action taken, no visual change needed |

## Accessibility

- All tool result cards use `role="article"` with `aria-label="[Entity Type] [action] result"`
- DeleteConfirmationCard uses `role="alertdialog"` with `aria-describedby` pointing to the confirmation body text
- "Confirm Delete" button has `aria-label="Confirm permanent deletion of [entity name]"`
- Tool loading indicators use `aria-live="polite"` and `role="status"`
- Entity name links within cards use standard anchor semantics with descriptive text
- Tab order within DeleteConfirmationCard: "Keep It" first, then "Confirm Delete" (prevent accidental destructive action)
