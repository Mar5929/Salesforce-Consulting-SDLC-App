1. Create `src/actions/team.ts` with server actions:

   **inviteTeamMember action:**
   - Zod schema: `{ projectId: string, email: string (email format), role: enum ProjectRole }`
   - Requires caller to be PM or SOLUTION_ARCHITECT (use requireRole)
   - Look up Clerk user by email (use Clerk Backend API: `clerkClient.users.getUserList({ emailAddress: [email] })`)
   - If user not found in Clerk, return error "User not found. They must sign up first."
   - Check if user is already a member (query ProjectMember with projectId + clerkUserId)
   - If already a member, return error "User is already a member of this project."
   - Create ProjectMember record with role, displayName (from Clerk), email, status ACTIVE
   - Send `inngest.send({ name: EVENTS.PROJECT_MEMBER_INVITED, data: { projectId, memberId, invitedBy: ctx.userId } })`
   - Return the created member

   **removeTeamMember action:**
   - Zod schema: `{ projectId: string, memberId: string }`
   - Requires caller to be PM or SOLUTION_ARCHITECT
   - Cannot remove yourself
   - Soft delete: update ProjectMember status to REMOVED, set removedAt to now()
   - Send audit event via inngest

   **updateMemberRole action:**
   - Zod schema: `{ projectId: string, memberId: string, newRole: enum ProjectRole }`
   - Requires caller to be PM or SOLUTION_ARCHITECT
   - Update ProjectMember role field
   - Send `inngest.send({ name: EVENTS.PROJECT_MEMBER_ROLE_CHANGED, data: { projectId, memberId, oldRole, newRole, changedBy: ctx.userId } })`
   - Return updated member

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | team management actions | mitigate | requireRole check enforces only PM and SOLUTION_ARCHITECT can invite/remove/change roles |
| T-03-02 | Tampering | project creation form | mitigate | Zod schema validation via next-safe-action on createProject action; all fields validated server-side |
| T-03-03 | Information Disclosure | project data in settings | mitigate | Project layout validates membership via getCurrentMember before rendering; scopedPrisma enforces projectId on all queries |
| T-03-04 | Tampering | role change action | mitigate | updateMemberRole validates caller has PM or SA role before changing target's role |
| T-03-05 | Denial of Service | team invite | accept | No rate limiting on invites in V1. Low risk: internal tool with <30 users. Consider rate limiting if needed. |
</threat_model>
