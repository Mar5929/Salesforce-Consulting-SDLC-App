<decisions>
## Implementation Decisions

### Question System UX
- **D-01:** Inline quick-add (Cmd+K or floating action button) for fast question creation, plus contextual creation from entity action menus (epic, feature, story) that auto-populates scope. Both paths available.
- **D-02:** Both kanban and table views with a toggle. Table is default (matches Linear list view aesthetic). Kanban shows columns per lifecycle status with drag-to-advance.
- **D-03:** Question lifecycle: Open > Scoped > Owned > Answered > Reviewed. Status badges in table, column headers in kanban.
- **D-04:** AI impact assessment shown as inline collapsible expansion below the answer. Badges for unblocked items, contradictions, and new questions raised. No separate panel needed.
- **D-05:** SA review flagging uses both inline warning flag on the question detail view AND a filtered "Needs Review" view for batch processing. Badge count in sidebar nav shows pending reviews.
- **D-06:** Question filtering and search via table column filters (status, category, scope, owner, priority) plus the global search bar.

### Transcript Processing Flow
- **D-07:** Dedicated transcript page (not modal) — transcripts are a primary workflow. Upload via drag-and-drop zone or paste.
- **D-08:** Multi-step AI processing presented as a chat session. AI extracts items, user confirms/edits at each checkpoint inline within the session.
- **D-09:** Extracted items displayed as grouped cards by type (questions, decisions, requirements, risks). Each card has accept/reject/edit actions.
- **D-10:** Deduplication shown inline — "Similar to existing Q-14" with merge or skip options per item.
- **D-11:** Processing runs as Inngest step function (TRNS-05). Status and results visible in the chat session interface (TRNS-06).

### Chat Interface
- **D-12:** General project chat accessible from sidebar nav (one per project). Task-specific sessions launched from context (e.g., "Process Transcript" button creates a session).
- **D-13:** Streaming responses via Vercel AI SDK `useChat` hook. All conversations persist with full message history.
- **D-14:** Split layout for task-specific sessions: message list on left, context panel on right showing related entities, token usage, and cost per session. General chat is full-width.
- **D-15:** Token/cost display as subtle footer metadata per AI message. Session total in the chat header.

### Knowledge Architecture
- **D-16:** Knowledge articles listed in a dedicated "Knowledge" section in sidebar. Staleness indicators as color-coded badges (green = fresh, yellow = aging, red = stale) based on last refresh time.
- **D-17:** Article detail view shows version history, source references (which questions/decisions/transcripts fed it), and related entities.
- **D-18:** Article refresh runs as Inngest background jobs triggered by `article/flagged-stale` event. Staleness detection triggered by `entity/content-changed` events.
- **D-19:** Two-pass context retrieval for AI: load article summaries first, then full content for most relevant articles (KNOW-05).
- **D-20:** AI proactively identifies discovery gaps and assesses readiness to start building (KNOW-06). Surfaced on the dashboard as "Recommended Focus" items.

### Search Experience
- **D-21:** Unified search bar at top of app (Cmd+K shortcut). Results shown across all three layers in grouped sections: filtered (structured), full-text (keyword), semantic (meaning-based).
- **D-22:** Search results ranked by relevance with source type badges. Results link directly to the relevant entity detail view.
- **D-23:** Embeddings generated by Inngest background jobs on entity create/update via `embedding/batch-requested` event.

### Discovery Dashboard
- **D-24:** Single discovery dashboard page accessible from sidebar. Sections: outstanding questions (by status/category), blocked items list, project health score card, AI "Current Focus" summary, AI "Recommended Focus" summary.
- **D-25:** Dashboard reads pre-computed cached data, not live queries. Inngest-triggered synthesis via `project/state-changed` event refreshes the cache.
- **D-26:** Health score derived from discovery completeness — percentage of questions answered, open risks, knowledge coverage gaps.

### Notifications
- **D-27:** Bell icon in app header with unread count badge. Dropdown panel shows recent notifications grouped by time.
- **D-28:** Click-through navigation from notification to the relevant entity. Mark read individually or bulk "mark all read."
- **D-29:** Notifications dispatched via Inngest background jobs triggered by `notification/send` event. 10 defined event types per spec.
- **D-30:** Priority-based notification ordering — high priority notifications surface first.

### AI Agent Harness
- **D-31:** Three-layer architecture as specified: task definitions > execution engine > context assembly. Task definitions specify required context, token budget, output schema, and validation rules.
- **D-32:** Execution engine manages AI calls with retry logic (3 retries, exponential backoff matching Phase 1 Inngest pattern), token tracking, and cost recording per session.
- **D-33:** Context assembly builds scoped context packages from multiple data sources within token budgets. Uses the two-pass retrieval strategy (summaries first, then full content).
- **D-34:** AI ambiguity handling is context-dependent per spec: ask inline when user present, best-guess + flag in task sessions, flag-only in background jobs (AGENT-05).
- **D-35:** All AI interactions logged with token usage, cost, and session linkage via SessionLog model.

### Claude's Discretion
- Exact Cmd+K command palette implementation details and keyboard shortcuts
- Kanban drag interaction specifics (optimistic updates, animation)
- Transcript processing checkpoint granularity (how many confirmation points)
- Chat message rendering details (markdown, code blocks, formatting)
- Knowledge article card design and density
- Search result ranking algorithm weights across the three layers
- Dashboard layout specifics (grid, card sizes, chart types for health score)
- Notification dropdown styling and animation
- Loading states, error boundaries, and empty state designs for all new views
- Exact token budget allocation across agent harness task types

## Specific Ideas

- Question creation should feel like Linear's Cmd+K quick issue creation — fast, minimal friction
- Contextual question creation from entities auto-populates scope (reduces manual field filling)
- Both kanban and table views for questions — user preference toggle, like Linear/Notion
- Transcript processing as a guided chat session rather than a black-box batch process — user stays involved at key decision points
