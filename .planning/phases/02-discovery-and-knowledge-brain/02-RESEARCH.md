# Phase 2: Discovery and Knowledge Brain - Research

**Researched:** 2026-04-06
**Domain:** AI agent harness, chat streaming, knowledge synthesis, semantic search, background jobs
**Confidence:** HIGH

## Summary

Phase 2 is the largest phase in the project -- 47 requirements across 8 subsystems (questions, agent harness, transcripts, chat, knowledge, search, dashboard, notifications). The technical spec (SESSION-3-TECHNICAL-SPEC.md) provides exceptionally detailed implementation guidance including full TypeScript implementations for the three-layer agent harness, tool definitions, search queries, dashboard queries, and Inngest step functions. The Prisma schema already has all Phase 2 entities deployed from Phase 1.

The primary technical risk is the embedding model choice. The schema uses `vector(1536)` which matches OpenAI's `text-embedding-3-small`, but the project spec says "Claude API only in V1." Anthropic does not offer embeddings -- they recommend Voyage AI, which defaults to 1024 dimensions. This needs resolution before the embedding pipeline can be built. All other infrastructure (Inngest events, Prisma models, shadcn components) is ready.

**Primary recommendation:** Build in dependency order: agent harness core (Layer 3 context assembly, Layer 2 engine, Layer 1 task defs) first, then chat infrastructure, then question system (which uses the harness for impact assessment), then transcript processing (which uses chat + harness + questions), then knowledge/search/dashboard/notifications. This respects the data flow: harness powers everything, chat renders harness interactions, questions are the atomic unit, transcripts produce questions/decisions, knowledge synthesizes from all of the above.

### Embedding Model (DECISION REQUIRED)

The Prisma schema defines `vector(1536)` for embedding columns. Anthropic does not provide an embeddings API. Two options:

| Option | Model | Dimensions | Pricing | Compatibility |
|--------|-------|-----------|---------|---------------|
| **A: OpenAI** | text-embedding-3-small | 1536 (native) | $0.02/MTok | Schema already matches. Requires `openai` npm package. |
| **B: Voyage AI** | voyage-3-large | 1024 default, 1536 configurable | ~$0.06/MTok | Anthropic-recommended. Requires `voyageai` npm package. Schema needs change to `vector(1024)` or dimension override. |

**Recommendation:** Use OpenAI `text-embedding-3-small`. The schema already uses 1536 dimensions. Cost is lower. Adding a second API key (OpenAI) for embeddings only is minimal friction. The embeddings are infrastructure, not the AI "brain" -- this doesn't conflict with the "Claude API only" decision which is about the reasoning/generation layer. ### Anti-Patterns to Avoid

- **Using Vercel AI SDK for agent harness backend:** The AI SDK abstracts away tool use iteration control. The agent harness needs direct control over message history, tool result handling, iteration counting, and session logging. Use `@anthropic-ai/sdk` directly.
- **Loading full context for every AI call:** Each task type has a specific context budget (Section 4.2 of tech spec). Context loaders must be scoped -- transcript processing does NOT load org components.
- **Using $queryRawUnsafe:** Banned in V1. All raw SQL must use tagged template literals for parameterization.
- **Using dangerouslySetInnerHTML without DOMPurify:** All AI-generated content rendered as HTML must go through DOMPurify.sanitize() first.
- **Making embedding generation synchronous:** Embeddings are always generated asynchronously via Inngest. The entity is created with `embeddingStatus: 'PENDING'` and search gracefully falls back to full-text.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Chat streaming | Custom SSE/WebSocket streaming | Vercel AI SDK `useChat` + `streamText` | Battle-tested streaming with automatic reconnection, abort handling, loading states |
| HTML sanitization | Regex-based tag stripping | isomorphic-dompurify | XSS prevention is deceptively complex, DOMPurify handles edge cases |
| Full-text search | Application-level text matching | PostgreSQL tsvector + GIN indexes | Database-level FTS handles stemming, ranking, highlighting natively |
| Semantic search | Custom similarity scoring | pgvector with HNSW indexes | Hardware-accelerated approximate nearest neighbor search |
| Background job orchestration | Custom queue with setTimeout | Inngest step functions | Automatic retries, checkpointing, concurrency control, observability |
| Command palette | Custom modal with search | shadcn Command (cmdk) | Accessible keyboard navigation, fuzzy matching, grouping built-in |
| Display ID generation | UUID-based or random IDs | generateDisplayId() with MAX query + retry | Human-readable, scoped, sequential -- better UX than opaque IDs |
| Input sanitization for AI content | Custom string escaping | DOMPurify + Prisma parameterized queries | Two layers of defense: prompt boundary + sanitization |

## Common Pitfalls

### Pitfall 1: Embedding Dimension Mismatch
**What goes wrong:** Schema defines `vector(1536)` but embedding model outputs different dimensions, causing pgvector insert failures.
**Why it happens:** Anthropic doesn't offer embeddings. Different providers use different default dimensions.
**How to avoid:** Choose embedding provider FIRST, then verify vector column dimensions match. If using Voyage AI (1024 default), either configure `output_dimension: 1536` on voyage-3-large or change schema to `vector(1024)`.
**Warning signs:** pgvector insert errors mentioning dimension mismatch.

### Pitfall 2: Chat Streaming vs Agent Harness Confusion
**What goes wrong:** Developer uses Vercel AI SDK for transcript processing or agent harness tasks, losing control over tool execution flow and session logging.
**Why it happens:** Two different AI libraries serve two different purposes -- easy to conflate.
**How to avoid:** Strict separation: `@anthropic-ai/sdk` for ALL agent harness tasks (transcript processing, impact assessment, dashboard synthesis, etc.). `ai` SDK for chat streaming UI ONLY.
**Warning signs:** Missing SessionLog entries, no tool use tracking, no token counting on agent tasks.

### Pitfall 3: tsvector Migration via Prisma
**What goes wrong:** Prisma doesn't natively support tsvector columns, triggers, or GIN indexes. Trying to manage these through the Prisma schema leads to migration failures.
**Why it happens:** tsvector and trigger functions require raw SQL that Prisma's schema language can't express.
**How to avoid:** Use Prisma `db execute` or a custom migration file to add tsvector columns, search triggers, and GIN indexes. These live outside the Prisma schema as manual SQL migrations. The tech spec (Section 8.1) provides the exact SQL.
**Warning signs:** Prisma migration errors, missing search_vector columns.

### Pitfall 4: Context Window Budget Overrun
**What goes wrong:** AI responses degrade or fail because too much context is loaded, exceeding model limits or wasting tokens.
**Why it happens:** Context loaders fetch too broadly. Easy to add "just one more" data source.
**How to avoid:** Each task definition specifies a target token budget (tech spec Section 4.2). Monitor via SessionLog.inputTokens. Start conservative, expand based on real usage data.
**Warning signs:** SessionLog showing consistently high input token counts, AI responses that don't use provided context.

### Pitfall 5: Inngest Event Flooding
**What goes wrong:** Processing a transcript that creates 15 entities triggers cascading events (entity.content-changed for each, which triggers staleness checks, which triggers article refresh, which triggers embedding generation...).
**Why it happens:** Event-driven architecture can create amplification loops.
**How to avoid:** Use batched events (tech spec Section 7.4.1). Emit ONE embedding.batch-requested event per agent invocation, not per entity. Staleness detection runs at end of harness execution, not per tool call. Debounce project.state-changed events.
**Warning signs:** Inngest dashboard showing event spikes, exceeding free tier limits quickly.

### Pitfall 6: Race Conditions in Display ID Generation
**What goes wrong:** Two concurrent question creations get the same display ID, causing unique constraint violations.
**Why it happens:** MAX query + increment is not atomic.
**How to avoid:** Wrap in a database transaction. The unique constraint acts as safety net -- catch the violation error and retry with the next number. At V1 scale (1-3 concurrent users per project), this is extremely rare but must be handled.
**Warning signs:** Prisma unique constraint violation errors during question/decision creation.

### Pitfall 7: Vercel Serverless Cold Starts on Inngest Functions
**What goes wrong:** Inngest function invocations time out or run slowly due to Vercel serverless cold starts.
**Why it happens:** Vercel serverless functions have cold start overhead. Prisma client initialization adds to this.
**How to avoid:** Use the existing Prisma client singleton pattern (src/lib/db.ts). Keep Inngest function handlers lightweight -- don't import unused modules. Vercel function timeout defaults to 10s on free tier -- ensure transcript processing steps individually complete within this window (each step.run() is a separate function invocation).
**Warning signs:** Inngest showing function timeouts, partial step completions.

```typescript
    concurrency: { limit: 10, scope: "env", key: "event.data.projectId" },
  },
  { event: EVENTS.NOTIFICATION_SEND },
  async ({ event, step }) => {
    await step.run("create-notification", async () => {
      // Determine recipients based on event type and project membership
      const recipients = await resolveRecipients(
        event.data.projectId,
        event.data.type,
        event.data.entityId
      );
      // Create notification records
      await prisma.notification.createMany({
        data: recipients.map(recipientId => ({
          projectId: event.data.projectId,
          recipientId,
          type: event.data.type,
          title: event.data.title,
          body: event.data.body ?? null,
          entityType: event.data.entityType,
          entityId: event.data.entityId,
        })),
      });
    });
  }
);
```

### Chat Route Handler with Context Assembly

```typescript
// Source: D-13 (Vercel AI SDK useChat), AGENT-04 (context assembly)
// app/api/chat/route.ts
import { anthropic } from '@ai-sdk/anthropic';
import { streamText } from 'ai';
import { getProjectSummary, getOpenQuestions, getArticleSummaries } from '@/lib/agent-harness/context';

export async function POST(req: Request) {
  const { messages, projectId } = await req.json();

  // Assemble context within token budget (~9-15K)
  const [summary, questions, articles] = await Promise.all([
    getProjectSummary(projectId),
    getOpenQuestions(projectId),
    getArticleSummaries(projectId),
  ]);

  const systemPrompt = buildChatSystemPrompt({ summary, questions, articles });

  const result = streamText({
    model: anthropic('claude-sonnet-4-20250514'),
    system: systemPrompt,
    messages,
    maxTokens: 4096,
    // Token tracking done separately via onFinish callback
    async onFinish({ usage }) {
      await logChatUsage(projectId, usage.promptTokens, usage.completionTokens);
    },
  });

  return result.toDataStreamResponse();
}
```

## Open Questions

1. **Embedding Model Choice**
   - What we know: Schema uses vector(1536). Anthropic has no embeddings API. Voyage AI (Anthropic-recommended) defaults to 1024 dimensions. OpenAI text-embedding-3-small natively outputs 1536.
   - What's unclear: Whether "Claude API only in V1" extends to embeddings or just the reasoning/generation layer.
   - Recommendation: Use OpenAI text-embedding-3-small for embeddings. The architectural boundary ("Claude for reasoning") is about the AI brain, not infrastructure. Embeddings are a utility function. If this feels like a violation, Voyage AI voyage-3-large can be configured to output 1536 dimensions but at higher cost.

2. **tsvector Migration Strategy**
   - What we know: Prisma can't express tsvector columns, triggers, or GIN indexes in its schema language.
   - What's unclear: Whether to use Prisma's `db execute` command, a custom SQL migration file, or a post-deploy script.
   - Recommendation: Create a custom Prisma migration (empty migration file, manually add SQL) for search infrastructure setup. This keeps it in the Prisma migration history.

3. **Vercel Function Timeout for Transcript Processing**
   - What we know: Each Inngest step.run() is a separate Vercel function invocation. Free tier timeout is 10s.
   - What's unclear: Whether complex agent loop iterations (Claude API call + tool execution + DB writes) complete within 10s.
   - Recommendation: Profile during development. Individual Claude API calls typically complete in 2-5s. If timeouts occur, either upgrade Vercel plan (Pro: 60s) or break steps into smaller units.

4. **Chat Context Strategy for General Chat**
   - What we know: PRD 8.2 says general chat is "stateless per-message" -- each message gets injected project context but no conversational memory. But useChat maintains local message history for display.
   - What's unclear: Whether to send the full message history to Claude (expensive but conversational) or just the latest message with injected context (cheap but each message is independent).
   - Recommendation: Start with injecting the last 10 messages as context (pragmatic middle ground). This provides conversational continuity without unbounded growth. The PRD's "stateless" language refers to the backend not maintaining persistent chat state beyond the Conversation model -- but the message history is still useful for Claude to understand conversational flow within a session.
