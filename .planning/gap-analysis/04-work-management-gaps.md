# Work Management Gap Analysis

**Domain:** Epic / Feature / Story Management
**PRD Sections Audited:** 5.2.1 (EpicPhase), 10 (Work Items), 15 (Guardrails)
**Audit Date:** 2026-04-08
**Auditor:** Code Explorer Agent

---

## Executive Summary

The work management domain is the most structurally complete area of the codebase. The hierarchy (Epic > Feature > Story), the 7-state status machine, role-gated transitions, and the EpicPhase grid are all implemented. The AI story generation pipeline is wired end-to-end (the `WORK-06` breakage documented in the milestone audit has been resolved). However, seven substantive gaps remain between what the PRD requires and what exists in code. The most critical are: (1) no mandatory field validation on the DRAFT → READY transition, meaning a story with no persona, no components, and no test cases can be marked Ready; (2) test case stubs are never auto-generated during story acceptance — the TestCase model exists but nothing writes to it from the story generation or creation path; (3) the PRD's dual-mode impacted components (free-text AND linked OrgComponent records) is only half-implemented — linked OrgComponent mode is present in the schema and context-package API but completely absent from the story form UI; (4) EpicPhase records are never auto-initialized when an epic is created; and (5) the six Salesforce development guardrails are entirely absent from story generation AI prompts, which is the web application's documented responsibility.

## Gap Inventory

### GAP-WORK-001: No mandatory field validation on DRAFT → READY transition

- **Severity:** Critical
- **Perspective:** Functionality
- **PRD Reference:** Section 10.3, 10.4
- **What PRD says:** "Every user story must have the following fields populated before it can be marked as 'Ready'... Stories in 'Draft' status can have incomplete fields. Only the transition to 'Ready' enforces full validation." The seven mandatory fields are: Persona, Description, Acceptance Criteria, Linked Epic or Feature, Estimated Story Points, Test Case(s), and Impacted Salesforce Components.
- **What exists:** The `updateStoryStatus` action in `src/actions/stories.ts` calls `canTransition(existing.status, parsedInput.status, member.role)` at line 282, which is a pure role-permission check from `src/lib/story-status-machine.ts`. There is no field-completeness check anywhere before or during the DRAFT → READY transition. A story with no persona, no description, no story points, no components, and no test cases can be transitioned to READY by any PM or SA.
- **The gap:** Zero mandatory field validation on the DRAFT → READY transition. The `canTransition` function is purely a role/direction check. No server action, no middleware, and no UI layer enforces the seven mandatory fields before Ready status is granted.
- **Scope estimate:** M
- **Dependencies:** GAP-WORK-004 (test cases must exist before they can be checked), GAP-WORK-005 (components validation requires deciding what "populated" means)
- **Suggested phase grouping:** Story Quality Enforcement

### GAP-WORK-005: Story components uniqueness constraint broken for free-text entries

- **Severity:** Significant
- **Perspective:** Architecture
- **PRD Reference:** Section 10.3 (Impacted Salesforce Components)
- **What exists:** The `StoryComponent` model has `@@unique([storyId, orgComponentId])` at line 1379 in `prisma/schema.prisma`. This constraint is designed for the linked-component mode (two stories cannot link to the same OrgComponent twice). For free-text entries, `orgComponentId` is `null`. In PostgreSQL, `NULL != NULL` in unique constraints — multiple rows with `orgComponentId = null` on the same story are permitted by the database. The application-level dedup in `addStoryComponent` (lines 373–385 of `src/actions/stories.ts`) queries `WHERE storyId = X AND componentName = Y AND impactType = Z AND projectId = P` which does catch free-text duplicates. However, there is no database-level constraint preventing duplicate free-text entries, and the `applyEnrichmentSuggestion` action in `src/actions/enrichment.ts` uses a raw `prisma.storyComponent.create` loop (lines 100–113) with no dedup check, meaning the enrichment path can create duplicate free-text component records.
- **The gap:** The enrichment path (`COMPONENTS` category in `applyEnrichmentSuggestion`) creates component records without checking for existing free-text duplicates. The database unique constraint only covers `orgComponentId` pairs, not free-text entries. Running AI enrichment twice on the same story will create duplicate component rows.
- **Scope estimate:** S
- **Dependencies:** None
- **Suggested phase grouping:** Story Quality Enforcement

## Schema-Level Observations

**`StoryComponent.componentName` is nullable in schema but treated as required in application code.** The Prisma model at line 1372 has `componentName String?`, but `addStoryComponent` requires `componentName: z.string().min(1)` and the `ComponentSelector` UI always provides a value. This nullable field was likely intended for the linked-OrgComponent mode (where `orgComponentId` is set and `componentName` could be derived from the org component's label). The dual-nullable design (`componentName?` OR `orgComponentId?`) means both can be null at the DB level — no constraint prevents a `StoryComponent` row with neither field set.

**`StoryComponent` unique constraint covers only `orgComponentId` path.** The `@@unique([storyId, orgComponentId])` constraint does not prevent duplicate free-text entries (both with `orgComponentId = null`, PostgreSQL treats NULLs as non-equal in unique constraints). Application-level dedup exists in `addStoryComponent` but not in `applyEnrichmentSuggestion`.

**EpicPhase `sortOrder` is a required non-nullable Int** in the schema but the upsert in `updateEpicPhaseStatus` only sets it on create (using `PHASE_SORT_ORDER[phase]`). This is correct design — sort order is fixed per phase type. No issue here.

## Additional Observations

**EpicPhase grid displays phantom data.** Due to GAP-WORK-004, new epics have zero `EpicPhase` rows. The grid's `getPhaseStatus()` fallback to `"NOT_STARTED"` makes this invisible to users, but any analytics or dashboard query against the `EpicPhase` table for a new epic returns nothing. The first cell-click per phase writes the record via upsert — so epics that have never been interacted with are undetectable from "all NOT_STARTED" in the grid but absent from the DB.

**AI-driven EpicPhase status transitions are not implemented.** PRD Section 5.2.2 states the AI updates EpicPhase records as stories within the epic progress through statuses. The `updateStoryStatus` action and all Inngest functions fire no EpicPhase update. The grid is entirely manually maintained.

**`StoryComponent` has a dual-nullable design risk.** Both `componentName` and `orgComponentId` are nullable, and no DB constraint prevents a row where both are null. Application-level code always provides one, but there is no schema-level guard.

**The story generation context loader does not include open (unanswered) questions.** `getStoriesContext` loads only `ANSWERED` questions. The PRD describes the AI as generating stories that cover requirements not yet addressed. Open questions that block requirements — the most important context for gap analysis — are invisible to the story generation AI.
